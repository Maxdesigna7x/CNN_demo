03 / El Origen Técnico

Next-Token Prediction

En su nivel matemático más puro, un modelo de lenguaje hace una única tarea: calcular el siguiente token más probable en base a un texto de entrada.

  • No comprende intenciones semánticas ni lógica de negocio.
  • Funciona como un autocompletador predictivo de alta capacidad.
  • Todo comportamiento agéntico complejo surge a partir de este cimiento.

04 / Salida Plana

Salida Cruda del Modelo

El modelo no ejecuta código de manera interna ni interactúa con tu hardware. Su única respuesta real es una cadena de texto continuo.

  • Genera tokens uno tras otro basándose en probabilidades secuenciales.
  • No existe el concepto de burbujas de chat o roles en la salida cruda.
  • La "inteligencia" inicial es puramente una simulación sintáctica bien entrenada.

06 / Abstracción de Chat

La interfaz de chat

La conversación fluida es una interfaz construida por la aplicación externa. Debajo del capó, el historial completo se formatea como una estructura de datos JSON limpia.

  • System: Define la personalidad y las reglas del asistente.
  • User: Representa los comandos de entrada del usuario humano.
  • Assistant: Respuestas previamente devueltas por el modelo.
System: Eres un asistente enfocado en resolver consultas de forma directa.
¿Qué es un JSON?
Un formato ligero de intercambio de datos estructurados para computadoras.
07 / Contexto Acumulado

Memoria Conversacional

El modelo no retiene memoria activa entre llamadas API separadas. La memoria se simula enviando toda la conversación de vuelta en cada interacción.

  • Cada turno de chat incrementa los tokens que enviamos al LLM.
  • El modelo re-procesa el hilo completo desde cero en cada ejecución.
  • La ventana de contexto actúa como el límite físico de almacenamiento del prompt.
Instrucción del Sistema 15 tokens
Turno 1: Mensaje del Usuario 25 tokens
Turno 1: Respuesta del LLM 45 tokens
Turno 2: Nuevo Mensaje 20 tokens
Contexto Total Enviado: 105 tokens → LLM
08 / Limitaciones

El primer límite crítico

Un LLM por sí solo es una base estática cerrada. Sus predicciones provienen exclusivamente de dos fuentes fijas de información.

  • Parámetros de entrenamiento: Congelados en la fecha de corte.
  • Ventana de contexto: Espacio de lectura de tamaño finito.
✕

Sin acceso a datos dinámicos o internos

El LLM no puede saber qué compraste hoy, ni consultar políticas internas actuales de tu empresa si no se las provees explícitamente.

ℹ

La Solución

Necesitamos construir una arquitectura que busque datos en tiempo real y los inyecte directamente en la ventana del prompt activo.

09 / Arquitectura de Datos

RAG: Inyección Externa

RAG (Generación Aumentada por Recuperación) no re-entrena al LLM. Busca fragmentos en bases de datos externas y los introduce dinámicamente en el prompt.

  • Recuperación: Encuentra chunks de texto relevantes en la base vectorial.
  • Aumentación: Modifica el prompt inyectando la información.
  • Generación: El LLM responde basándose en la evidencia provista.
Paso 1: Pregunta del Usuario
"¿Cuánto gasté el mes pasado en productos de comida?"
Paso 2: Búsqueda Vectorial (Fragmentos recuperados)
Relevancia: 12%
Gastos en transporte: $50
Relevancia: 95%
Gastos en comida: $120
Relevancia: 28%
Gastos en servicios: $80
Paso 3: Prompt Aumentado + Respuesta Generada
"Pregunta: ¿Cuánto gasté en comida? Contexto: Gastos comida = $120.
[Respuesta LLM]: Gastaste un total de $120 en comida el mes pasado."
10 / Calidad de Respuesta

Cómo cambia la respuesta

Con RAG, el modelo pasa de adivinar o alucinar a responder con datos reales verificados directamente sobre tu documentación.

  • Sin RAG: Respuestas genéricas, especulativas o desactualizadas.
  • Con RAG: Precisión fundamentada en fuentes externas inyectadas.

Sin RAG

"No puedo acceder a tus transacciones en tiempo real. Asumo que el gasto promedio en comida suele rondar los $100..."

Con RAG

"De acuerdo con tus registros de transacciones del mes pasado, gastaste un total de $120 en comida."
11 / Limitaciones Avanzadas

El segundo límite crítico

El modelo ahora tiene acceso a información externa, pero se enfrenta a una barrera de acción: sigue produciendo únicamente texto.

  • Puede sugerir y redactar una transferencia bancaria perfecta.
  • Puede escribir el código exacto para enviar un correo de soporte.
  • But **carece de ejecución autónoma**: el LLM por sí solo no puede realizar clics ni conectarse a protocolos externos.
✕

El modelo aconseja, pero no ejecuta

Recomendar una acción en texto no es lo mismo que ejecutarla. Necesitamos un traductor de intenciones que las máquinas puedan procesar.

12 / Control de Output

Structured Output & Tools

Alineamos al modelo para emitir esquemas de datos JSON validados en vez de prosa natural, permitiendo que otros sistemas interpreten su intención como un comando formal.

  • El modelo no corre la función directamente.
  • El modelo emite una solicitud estructurada con argumentos explícitos para que el orquestador externo la ejecute.

La pieza invisible del sistema

El orquestador es el software tradicional que lee los comandos estructurados del modelo y los ejecuta en el mundo real.

🔍

🔄

14 / Bucles Agénticos

El Ciclo de Acción

El verdadero comportamiento agéntico surge cuando permitimos al sistema iterar en un bucle continuo de toma de decisiones y ejecución.

  • Observar: El sistema evalúa el estado del contexto actual.
  • Decidir: El modelo determina si necesita usar una herramienta.
  • Actuar: El orquestador ejecuta la acción y devuelve la observación.

El Harness Agéntico

Seguridad y Control

Un Harness Agéntico es el entorno de ejecución controlado (sandboxed) y estructurado que envuelve al modelo de lenguaje. Es la infraestructura de software encargada de transformar las intenciones en texto del LLM en acciones seguras, validadas y auditables.

Context (Contexto)

Construye y actualiza la ventana dinámica de tokens inyectando historial y datos en tiempo real.

Tools (Herramientas)

Catálogo y esquemas de funciones que el modelo tiene permitido solicitar bajo validación estricta.

State (Estado)

Monitorea y almacena las variables operativas y de sesión a lo largo de ciclos de ejecución complejos.

Memory (Memoria)

Persistencia semántica externa para recordar datos históricos más allá de la ventana de chat.

Validation (Validación)

Filtra y sanea todos los outputs JSON del modelo antes de ejecutar cualquier acción en producción.

Execution (Ejecución)

Entorno aislado (Sandbox) donde se ejecutan las llamadas API y tareas lógicas de forma segura.

16 / Anatomía del Flujo

Funcionamiento del Harness en Acción

1. Entrada & Configuración

El harness configura el System Prompt limitando las capacidades y estructura la lista de herramientas válidas con JSON Schema.

2. Prompt de Usuario

El mensaje inicial entra al flujo buscando y sumando contexto dinámico mediante RAG.

3. Ejecución Aislada

El orquestador intercepta la llamada, valida argumentos, ejecuta la acción real y formatea el resultado final.

"Eres un agente financiero. Tienes acceso a la función: transferir_fondos(destino, monto)"
User Prompt
"Envía 150 dólares de mi cuenta de ahorros a la cuenta de María."

[VALIDADO] Llamando API Bancaria...
Resultado: {"status": "success", "tx_id": 89432}
Final Response
"He completado la transferencia de $150 a María con ID de transacción 89432."
17 / Resiliencia en Producción

Retos y Buenas Prácticas

Desplegar sistemas agénticos seguros requiere asumir que los modelos cometerán errores de forma continua.

  • Los LLMs suelen alucinar herramientas que no existen o argumentos incorrectos.
  • Es crítico limitar el consumo de tokens y llamadas consecutivas.
  • La trazabilidad completa y auditable es indispensable.
Modos de Fallo
  • Bucles infinitos de llamadas recursivas
  • Alucinación de herramientas no configuradas
  • Argumentos Json con sintaxis rota o inválida
Estrategias de Control
  • Límites estrictos de pasos de ejecución (Max Steps: 5)
  • Validación estricta de esquemas JSON Schema
  • Aislamiento en Sandbox para código generado
